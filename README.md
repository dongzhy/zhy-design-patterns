##⼯⼚⽅法模式的优缺点
#优点
解耦了对象创建和使⽤：调⽤⽅不再关⼼具体是怎么 new 出来的对象，只负责使⽤，这样⼀来，如果后续对
象的创建逻辑变了，也不会影响到业务逻辑，代码更稳定、更灵活。
符合开闭原则，易于扩展：当需要新增⼀个产品时，只需要新增⼀个对应的⼯⼚类就⾏，原来的代码不⽤
动。这对于后期产品种类多、变化快的场景⾮常友好。
提⾼了代码的复⽤性和可维护性：创建逻辑集中在⼯⼚⾥，避免了重复造轮⼦。有什么调整，只需要改⼀处
地⽅，维护成本低，也更容易统⼀管理。
#缺点
类的数量会增多：每新增⼀种产品，通常就要新增⼀个对应的⼯⼚类。产品多了之后，⼯⼚类也多，项⽬结
构可能会显得臃肿。
增加了系统复杂度：相较于直接 new，⼀个⼯⼚类、⼀个产品接⼝这样组合起来的结构，对于⼩项⽬或者逻
辑⽐较简单的场景来说，可能会显得有点过于复杂。
有时候可能出现过度设计：如果产品结构本⾝很简单，⽤⼯⼚⽅法模式反⽽会让代码变得啰嗦，不如直接⽤
简单⼯⼚来得⾼效直接。


## 单例模式使用场景
# 全局唯⼀ID⽣成器：在电商、内容平台等⾼并发系统中，需要全局唯⼀的订单号、⽤户ID、消息ID等，通常
会实现⼀个雪花算法（Snowflake）或时间戳+随机数⽣成器，作为单例使⽤，保证线程安全和唯⼀性。
# 系统配置管理类：很多应⽤在启动时会加载⼀份配置（如系统参数、限流配置、第三⽅服务地址等），通过单
例模式保证在内存中只加载⼀次，全局共享，避免重复读取和资源浪费。
# 应⽤级缓存组件：例如⼀些基础数据（地区列表、标签分类、活动配置等）缓存到内存中，需要多个模块共
享，通常封装成单例类，提供统⼀的读写接⼝，保证数据⼀致性。
## 单例模式的优缺点
# 优点
全局唯⼀，资源可控：单例模式可以保证某个类在系统中始终只有⼀个实例，这对⼀些需要全局共享资源的
场景，⽐如配置管理器、线程池、缓存等，⾮常实⽤，既节省资源⼜避免冲突。
提供统⼀访问点：通过统⼀的 getInstance() ⽅法获取实例，不⽤每次都去 new，代码更简洁，管理也更集
中，有时候还能结合懒加载，提升性能。
便于扩展为多线程安全版本：只要设计得当，⽐如加锁、使⽤双重检查或静态内部类⽅式，都能很好地保证
线程安全，让单例在多线程环境下也能稳定运⾏。

1、资源管理统⼀
像 Runtime 这样的系统⼯具类，如果每次都 new ⼀份会造成资源浪费。Java 把它做成了单例，确保整个 JVM ⾥只
有⼀个实例，全局统⼀调度。Spring 中默认的 singleton 作⽤域也是⼀样，所有请求都会拿到同⼀个 Bean，实现
了资源的集中管理。
2、减少内存消耗
如果每次请求都 new ⼀个对象，尤其在并发场景下，系统负担会⾮常⼤。⽤单例模式，只初始化⼀次，后续复⽤，能
显著降低内存占⽤和对象创建的开销。
3、保证全局状态⼀致
像配置中⼼、⽇志管理器这种组件，必须保证状态统⼀。如果每次都有新对象，配置就可能不⼀致。单例可以确保整
个系统中使⽤的是同⼀个状态副本，不容易出错。
# 缺点
不利于测试和扩展：单例⼀旦写死了，很难在单元测试时替换成 mock 对象，也不容易在运⾏时切换成别的
实现，不利于灵活扩展和测试隔离。
可能引发隐藏的状态问题：由于单例在多个地⽅被共享使⽤，容易导致不⼩⼼在⼀个地⽅修改了它的状态，
影响到其他地⽅的⾏为，尤其在多线程环境下更容易出问题。
⽣命周期不可控：单例通常在 JVM ⽣命周期内都存在，不容易销毁，容易导致资源⽆法及时释放。如果单例
中持有了重资源，⽐如数据库连接或者⽂件句柄，可能造成内存泄漏⻛险。


## 框架中应用
# jdk
java.lang.runtime类就是标准的单例，通过getruntime方法返回唯一实例
# spring框架
默认情况下，自己写的bean，如果没有额外指定scope，都是单例
@Configuration
public class AppConfig {
@Bean
public UserService userService() {
return new UserService();
}
}
只要这个配置类被 Spring 扫描到，Spring 就会在容器初始化时创建⼀个 UserService 实例，放进它的 Bean 容器
⾥。之后我们每次调⽤ getBean(UserService.class) ，拿到的都是同⼀个对象。
